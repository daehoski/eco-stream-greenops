1인 설계 및 구현
Problem

데이터센터 전력의 30%는 좀비 서버에서 낭비됩니다

퍼블릭 클라우드는 서버리스(Serverless)로 해결했지만, 온프레미스 베어메탈 환경에서는 트래픽과 무관하게 서버가 24시간 가동됩니다. 유휴 서버 1대의 대기 전력은 약 300W, 연간 2,628kWh를 낭비합니다. Kubernetes의 HPA는 파드 수만 조절할 뿐, 물리 서버 전원은 제어하지 못합니다.

Solution

트래픽이 없으면 물리 서버를 실제로 꺼버립니다

Kafka의 버퍼링 + KEDA의 이벤트 기반 스케일링 + 커스텀 GreenOps 컨트롤러의 물리 노드 전원 제어를 결합하여, 유휴 상태의 Worker Node 전력을 0W로 만듭니다. 새 요청이 들어오면 Wake-on-LAN으로 수 초 내 부팅, 즉시 워크로드 처리를 시작합니다.

Core Implementation
Kafka KRaft

비동기 버퍼링

  • ZooKeeper 없이 Kafka 자체 합의(KRaft)
  • 10개 파티션으로 병렬 소비 구조
  • Worker 0개여도 메시지 유실 없음
  • Consumer Group 기반 파티션 할당
  • 댐(Dam) 패턴: 트래픽 폭주 흡수
KEDA

이벤트 기반 스케일링

  • CPU 기반 HPA 대비 실제 작업량 기반
  • lagThreshold: 1 (메시지 1개 = 즉시 반응)
  • minReplicaCount: 0 (완전 제로 스케일)
  • maxReplicaCount: 10 (파티션 수 매칭)
  • cooldownPeriod: 30s (안정적 스케일인)
GreenOps Controller

물리 노드 전원 제어

  • Bash FSM(상태 머신) 직접 구현
  • Pending Pod 감지 → WOL 패킷 전송
  • Empty Node 감지 → Cordon + SSH Shutdown
  • 10초 간격 Health Check Loop
  • Descheduler 연동 (빈 노드 생성)
GreenOps Lifecycle
1

IDLE → SCALE-OUT

사용자가 영상 업로드 → Kafka Lag 발생 → KEDA가 Worker Pod 0 → 10 생성

2

SCALE-OUT → WAKE-UP

Pod가 Pending 상태 → Controller가 k8s-worker2에 Wake-on-LAN 전송 → 노드 부팅 → Uncordon

3

PROCESSING

10개 Worker가 Kafka 파티션에서 병렬 소비 → FFmpeg 영상 처리 → Lag 점진적 감소

4

SCALE-IN → SHUTDOWN

Lag 0 도달 → 30초 쿨다운 → Pod 10 → 0 → Descheduler가 빈 노드 생성 → Controller가 Shutdown

Troubleshooting Experience
Symptom Pod 10개 중 3개만 작업 처리, 나머지 7개는 idle 상태로 큐를 소비하지 않음
Root Cause Kafka 토픽 파티션 3→10 변경 후, Consumer가 stale metadata를 유지하여 기존 3개 파티션만 소비. 신규 7개 파티션의 메시지가 적체
Resolution kubectl rollout restart deployment eco-web eco-worker — Consumer 재시작으로 메타데이터 리프레시, 10개 파티션 전체에서 균등 소비 확인
Symptom KEDA가 Lag 높다고 보고하지만, 실제 Worker들은 idle 상태. Pod가 0으로 내려가지 않음
Root Cause Consumer Group offset이 stale 파티션을 참조 → Kafka가 phantom lag를 보고 → KEDA가 불필요한 replica 유지
Resolution Deployment rolling restart로 모든 Consumer가 최신 offset으로 재합류 → Lag 0 확인 → 정상 scale-in 진행
